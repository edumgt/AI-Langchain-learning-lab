# 샘플 문서: IT 업무 기술 안내 (학습용)

## 헬프데스크 운영 원칙
- 장애 접수는 티켓 시스템(예: Jira Service Management, Zendesk)을 통해 단일 창구로 관리합니다.
- 우선순위 분류: P1(서비스 전면 중단) → P2(주요 기능 장애) → P3(성능 저하) → P4(일반 문의)
- P1·P2는 30분 이내 최초 대응, P3는 4시간, P4는 1영업일을 SLA 목표로 설정합니다.
- 해결 후 근본원인분석(RCA) 문서를 작성하고 재발 방지 대책을 티켓에 기록합니다.

## 네트워크 및 인프라 장애 대응
- 네트워크 장애 1차 점검: 케이블 연결 상태 → 스위치/AP 재시작 → DHCP 할당 확인 → DNS 조회 테스트
- 서버 응답 없음: ping → traceroute → 포트 상태 확인(netstat / ss) → 로그 확인(/var/log/syslog)
- VPN 접속 오류: 클라이언트 로그 확인 → 자격증명 재입력 → 방화벽 정책 검토 → 서버 측 인증서 만료 여부 확인
- 스토리지 가용량 경고: df -h 확인 → 대용량 파일 탐색(du -sh) → 로그 로테이션 설정 검토

## 보안 운영
- 패치 관리: 운영체제 및 주요 미들웨어는 월 1회 정기 패치, 긴급 취약점(CVSS 9.0 이상)은 즉시 패치
- 계정 관리: 퇴직자 계정은 퇴직 당일 비활성화, 권한 최소화 원칙(Least Privilege) 준수
- MFA(다단계 인증): 원격 접속(VPN, 클라우드 콘솔) 및 관리자 계정에 필수 적용
- 보안 사고 발생 시: 격리(네트워크 차단) → 증거 보전(로그 수집) → 보고 → 복구 순서로 진행

## 클라우드 운영 (AWS/Azure/GCP 공통)
- IAM 원칙: 루트 계정 사용 금지, 역할 기반 접근 제어(RBAC), 자격증명 90일 주기 교체
- 비용 최적화: 미사용 리소스(유휴 인스턴스, 미연결 EBS 등) 월간 점검 후 삭제
- 백업: 중요 데이터는 3-2-1 규칙(복사본 3개 / 매체 2종 / 오프사이트 1개) 적용
- 모니터링: CloudWatch / Azure Monitor / Cloud Logging으로 CPU·메모리·디스크 알람 설정

## 소프트웨어 개발 및 배포
- 브랜치 전략: main(운영) / develop(통합) / feature/* / hotfix/* 구조 권장(GitFlow)
- CI/CD: Pull Request 머지 시 자동 빌드·테스트(GitHub Actions, GitLab CI) → 스테이징 배포 → 운영 배포
- 코드 리뷰: 최소 1인 승인 필수, 보안·성능·가독성 체크리스트 활용
- 배포 전 체크리스트: DB 마이그레이션 롤백 계획, 환경변수 확인, 헬스체크 엔드포인트 동작 검증

## 데이터베이스 운영
- 정기 백업: 트랜잭션 로그는 1시간 이내, 전체 백업은 일 1회 수행
- 쿼리 성능: EXPLAIN / EXPLAIN ANALYZE로 실행계획 분석, 슬로우 쿼리 로그 활성화
- 인덱스 관리: 자주 조회되는 컬럼에 인덱스 추가, 미사용 인덱스 정기 정리
- 장애 복구: RTO(목표 복구 시간)·RPO(목표 복구 지점) 정의 후 분기별 복구 훈련 실시

## IT 서비스 관리 (ITSM)
- 변경 관리: 운영 환경 변경은 CAB(변경자문위원회) 승인 후 진행, 롤백 계획 필수
- 문제 관리: 반복 장애는 Known Error로 등록하고 임시 회피책(Workaround) 문서화
- 서비스 수준 관리(SLM): SLA/OLA/UC 정기 검토, KPI 대시보드 유지
- 자산 관리: 하드웨어·소프트웨어 라이선스 인벤토리 최신화, EOL(지원 종료) 일정 추적

## 사용자 교육 및 지원
- 비밀번호 정책: 영문·숫자·특수문자 조합 8자 이상, 90일 주기 변경, 이전 비밀번호 재사용 금지
- 피싱 예방: 의심 메일 첨부파일 클릭 금지, 발신자 도메인 확인, 내부 신고 채널 활용
- 원격 지원: 공식 원격 지원 도구(예: TeamViewer, AnyDesk)만 사용, 실행 전 사용자 동의 필수
- 소프트웨어 설치: IT 승인된 화이트리스트 소프트웨어만 설치, 무단 설치 시 보안 정책 위반
